# Story 1.1 — Identificação do Atendente nas Conversas Capturadas

## Descrição

Como analista do ML Laboratory,
quero que cada sessão de conversa registre qual atendente estava operando o WhatsApp,
para que as análises dos agentes de ML (conversation-analyst, behavioral-profiler, etc.) não misturem dados de atendentes diferentes e produzam insights precisos por pessoa.

## Contexto

Atualmente a tabela `ml_captura.sessoes_conversa` registra apenas o contato (cliente), mas não o atendente que estava do lado da empresa. Sem esse vínculo, análises de performance e perfil comportamental ficam distorcidas.

Existem dois modelos de operação:

- **Mono-agente:** 1 número WhatsApp → 1 pessoa fixa. O número já identifica o atendente.
- **Multi-agente:** 1 número → N atendentes (ex: Redrive, outras plataformas). A plataforma identifica o atendente via `identificador_externo` no payload do webhook.

## Escopo

### IN (o que será feito)
- Adicionar campo `tipo` (`mono` | `multi`) em `_plataforma.numeros_projeto`
- Adicionar campo `agente_default_id` (FK → `agentes_humanos`) em `_plataforma.numeros_projeto` — preenchido apenas quando `tipo = mono`
- Adicionar campo `agente_humano_id` (FK → `agentes_humanos`) em `ml_captura.sessoes_conversa`
- Criar migration idempotente para as 3 alterações acima
- Ajustar workflow n8n de captura (`ML-CAPTURA`) para resolver e preencher `agente_humano_id` automaticamente:
  - Se `tipo = mono`: usar `agente_default_id` do número
  - Se `tipo = multi`: extrair `identificador_externo` do payload → buscar `agente_humano_id` correspondente

### OUT (fora do escopo)
- Alteração na UI do Appsmith
- Alteração no fluxo de onboarding (formulário)
- Retroativo em sessões já capturadas
- Criação de novos agentes de ML

- [ ] **AC6:** Quando `identificador_externo` do webhook não bate com nenhum `agentes_humanos` cadastrado, o campo fica `null` e a sessão é salva normalmente (sem erro)

```sql
  id UUID PK,
  numero_id UUID FK → numeros_projeto,
  projeto_id UUID FK → projetos,
  nome VARCHAR(255),
  identificador_externo VARCHAR(255),  -- ID/nome usado pelo Redrive
  email VARCHAR(255),
  ativo BOOLEAN DEFAULT true,
  total_conversas INTEGER DEFAULT 0
)
```

### Exemplo de payload multi-agente (Redrive)
O campo `identificador_externo` deve ser buscado no `metadados JSONB` do webhook — confirmar com o usuário qual campo do payload identifica o atendente no Redrive.

- Não retroagir em sessões existentes (AC6)
