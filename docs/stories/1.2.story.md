# Story 1.2 — Suporte a Respondent Type e Endpoint de Integração Externa

## Descrição

Como plataforma ML Laboratory,
quero receber dados de atendimento de sistemas externos (ex: EsteticaIA) via webhook dedicado,
para que conversas de clínicas parceiras sejam capturadas, identificadas por tipo de respondente (IA, humano, especialista) e processadas pelos agentes de ML sem distorção.

## Contexto

A EsteticaIA propôs integração enviando dados de atendimento WhatsApp para o ML. O schema atual não suporta:
- Direção da mensagem por registro (incoming/outgoing)
- Tipo de respondente por mensagem (ai | human | specialist)
- Nome do respondente por mensagem
- Endpoint dedicado para payloads externos (formato diferente do Evolution API)

A Story 1.1 adicionou `agente_humano_id` nas sessões — esta story complementa adicionando granularidade por mensagem.

## Escopo

### IN
- Adicionar `direction`, `respondent_type`, `respondent_name` em `ml_captura.mensagens_raw`
- Criar migration 017 idempotente
- Criar workflow n8n dedicado `ML-EXTERNAL-esteticaia.json` com endpoint `/webhook/ml/external/esteticaia`
- Normalizar payload EsteticaIA → formato interno ML
- Resolver `agente_humano_id` a partir do `respondent_id` padronizado (ai:{slug} | human:{slug} | specialist:{slug})

### OUT
- Alterações nos workflows de análise ML downstream (Story 1.3 futura)
- Autenticação/segurança do endpoint (pode ser story separada)
- Retroativo em conversas existentes

```json
  "remote_jid": "5511999999999@s.whatsapp.net",
  "body": "Quero agendar uma avaliação",
  "timestamp": "2026-04-19T10:00:00Z",
  "type": "incoming",
  "respondent": "ai",
  "respondent_id": "ai:sofia-sdr",
  "attendant_name": "Sofia (IA)",
  "whatsapp": "5511988887777",
  "attendance_type": "multi",
  "sector": "comercial",
  "session_id": "studio-bella-5511999-20260419",
  "message_type": "text",
  "audio_url": null
}
```

### Mapeamento de campos
| Campo EsteticaIA | Campo ML interno |
|---|---|
| instanceName | instancias_evolution.instance_name |
| messageId | mensagens_raw.message_id |
| remote_jid | mensagens_raw.remote_jid |
| body | mensagens_raw.conteudo_raw |
| timestamp | mensagens_raw.created_at |
| type | mensagens_raw.direction |
| respondent | mensagens_raw.respondent_type |
| attendant_name | mensagens_raw.respondent_name |
| message_type | mensagens_raw.tipo |
| session_id | sessoes_conversa.metadados.external_session_id |

### Lógica de resolução de respondent_id
```
respondent_id = "ai:sofia-sdr"    → agente_humano_id = null
respondent_id = "human:maria"     → busca agentes_humanos por identificador_externo = "human:maria"
respondent_id = "specialist:dr-x" → busca agentes_humanos por identificador_externo = "specialist:dr-x"
```
