# Sessão 2026-04-28_22-47 — Railway Build Fix + Parceiros API

## O que foi feito

### 1. Correção do build Railway — `Module not found: @/lib/db`
- Diagnóstico: nixpacks com dois `package.json` (monorepo) causava WORKDIR ambíguo
- Fix: criado `portal-next/Dockerfile` multi-stage (node:20-alpine: deps → builder → runner)
- `portal-next/railway.json`: `builder: NIXPACKS` → `builder: DOCKERFILE`
- Commit: `11da7c4`

### 2. Correção Resend lazy init — `Missing API key`
- Causa: `new Resend(process.env.RESEND_API_KEY)` no nível do módulo executava durante "Collecting page data" do Next.js build (sem env vars de runtime)
- Fix: instanciação movida para dentro de `enviarEmail()`
- Commit: `8a5d94f`

### 3. Correção diretório `public/` ausente
- Causa: `COPY --from=builder /app/public ./public` falhava porque portal-next não tem assets estáticos
- Fix: criado `portal-next/public/.gitkeep`
- Commit: `dcae3ce`

### 4. Correção `schema_prefix` NOT NULL
- Diagnóstico via psycopg2 direto no banco Railway: INSERT falhava com `null value in column "schema_prefix" violates not-null constraint`
- Fix: `schema_prefix` adicionado ao INSERT usando o slug do parceiro
- Commit: `d0e0533`

## Erros resolvidos

| Erro | Causa | Fix |
|------|-------|-----|
| `Module not found: Can't resolve '@/lib/db'` | nixpacks WORKDIR ambíguo | Dockerfile explícito |
| `Missing API key. Pass it to constructor new Resend(...)` | Resend init em module scope | Lazy init no handler |
| `/app/public: not found` | public/ ausente no projeto | public/.gitkeep |
| `null value in column schema_prefix` | NOT NULL sem default | slug como schema_prefix |

## Decisões tomadas

- Dockerfile multi-stage em vez de nixpacks: controle total sobre WORKDIR e COPY
- Resend lazy init: padrão correto para Next.js — não instanciar serviços no module scope
- slug = schema_prefix para parceiros externos (único, consistente com padrão do projeto)
