## O que foi feito

### 1. Correção da falha silenciosa no envio de email/WhatsApp (commit 55c2eb8)

**Bug:** funções `enviarEmail` e `enviarWhatsApp` em `portal-next/app/api/admin/parceiros/route.ts` faziam `try/catch` mas só logavam o erro — não retornavam sinalização. A `Promise.all` no handler não recebia falha, e a tela de sucesso afirmava "E-mail e WhatsApp enviados" mesmo quando algum dos dois falhava silenciosamente.

**Causa secundária:** SDK do Resend não lança exceção em alguns erros (ex: domínio não verificado) — retorna `result.error` no objeto de resposta. O código antigo só capturava exceções, perdia esses casos.

**Correção:**
- Funções agora retornam `{ ok: true } | { ok: false, error: string }`
- `enviarEmail` verifica `result.error` do Resend explicitamente
- `enviarWhatsApp` reporta status HTTP da Evolution + corpo da resposta em caso de falha
- Validação de envs ausentes (`RESEND_API_KEY`, `RESEND_FROM`, `ONBOARDING_*`) faz parte do retorno
- Handler `POST /api/admin/parceiros` coleta os 2 resultados e expõe `email_status`, `email_error`, `whatsapp_status`, `whatsapp_error` no response
- UI `/admin/parceiros/novo` lê os novos campos e mostra ✅/⚠️ por canal com mensagem de erro específica
- Texto do email atualizado: removida menção a "credenciais Evolution" (fluxo antigo) → agora descreve "endpoint + API Key gerada pelo seu sistema"

**Arquivos modificados:**
- `portal-next/app/api/admin/parceiros/route.ts` (refatoração completa de envio + response)
- `portal-next/app/admin/parceiros/novo/page.tsx` (tipo do `result` + render condicional por canal)

**Build local validado:** `npm run build` em `portal-next/` passou limpo (todas as rotas compiladas, 0 erro de tipo).

**Commit + push:** `55c2eb8 fix(onboarding): reporta status real de email/whatsapp na criação de parceiro` → `42504a2..55c2eb8 main -> main` (Railway vai rebuildar `portal-ml` automaticamente).

## Decisões tomadas (registradas)

- **fonte = 'external_partner'** para mensagens vindas via webhook do parceiro — distingue origem técnica do canal direto Evolution sem quebrar queries existentes que filtram por `fonte = 'whatsapp'`. Reasoning: a empresa já é identificada por `projeto_id`, o atendente por `agente_humano_id`/`respondent_type`. `fonte` é canal técnico de entrada, não identificador de empresa nem atendente.
- **Mapeamento payload → mensagens_raw** definido (tabela completa documentada na resposta antes do `/fechar-sessao`):
  - `mensagem.id` → `message_id` (UNIQUE → dedupe)
  - `conversa_id` → `session_id`
  - `lead_id` → `remote_jid` (fallback `conversa_id`)
  - `mensagem.origem`: LEAD→incoming/unknown, ATENDENTE→outgoing/human, SISTEMA→unknown/ai
  - `mensagem.enviada_em` → `created_at`
  - Resto do payload → `metadados` jsonb
- **Validação Bearer:** comparação igualdade exata contra `_plataforma.projetos.webhook_api_key` (sem hash, fase 1). Slug não existe → 404, Bearer ausente/inválido → 401, payload incompleto → 400.
- **Audio sem transcrição automática no MVP** (recomendado, aguarda confirmação): salva `audio_url` mas não chama Groq Whisper — transcrição em batch depois.
- **agente_humano_id NULL no MVP** (recomendado, aguarda confirmação): salva `atendente_externo_id` em `metadados` para resolução futura.

## Erros resolvidos

- `Module not found '/tmp/ml-captura-current.json'` — `/tmp/` no bash do Windows não persiste entre subshells. Solução: salvar exports em `tmp/` no diretório do projeto e adicionar `tmp/` ao `.gitignore`.
- `column "criado_em" does not exist` — coluna em `_plataforma.projetos` é `created_at` (snake_case inglês), não `criado_em`.
- `relação "_plataforma.mensagens_raw" não existe` — schema correto é `ml_captura`, não `_plataforma`. Mensagens ficam em `ml_captura.mensagens_raw`.
