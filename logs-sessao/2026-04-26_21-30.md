# Sessão 2026-04-26 — Deploy Portal Railway + Fix Identificação de Remetentes

## O que foi feito

### 1. Fix — Identificação de remetentes nas conversas
- **Problema:** Mensagens na tela de conversas apareciam todas do mesmo lado sem indicar quem era o remetente
- **Causa raiz 1:** Campos `direction` e `respondent_type` em `ml_captura.mensagens_raw` estão todos como `"unknown"` — nunca foram preenchidos pelo n8n
- **Causa raiz 2:** Código usava `COALESCE(direction, 'incoming')` → todas as mensagens viravam `incoming` → lado esquerdo
- **Solução:** Usar `metadados.data.key.fromMe` (campo nativo do payload WhatsApp/Evolution API):
  - `fromMe: true` → bot/agente (direita, violeta) — usa `pushName` como label (ex: "Kátia - Cosmobeauty")
  - `fromMe: false` → contato (esquerda, cinza) — usa `pushName` como label (ex: "Van Sanches")
- **Arquivos alterados:** `portal-next/app/p/[slug]/conversas/[sessao_id]/page.tsx`
- **Volume de dados:** 524 mensagens `fromMe=true`, 705 `fromMe=false`, 1 null

### 2. Deploy do Portal Next.js no Railway
- **Serviço criado:** `portal-ml` no projeto `ml-laboratory`
- **URL de produção:** `https://portal-ml-production.up.railway.app`
- **Problema no build:** `Module not found: Can't resolve '@/lib/db'`
  - **Causa:** `tsconfig.json` sem `"baseUrl": "."` — o webpack do Railway não resolvia o alias `@/*` em build limpo
  - **Fix:** Adicionado `"baseUrl": "."` no `portal-next/tsconfig.json`
- **railway.json criado** em `portal-next/` com builder Nixpacks e healthcheck
- **Variáveis de ambiente** já configuradas no Railway (DATABASE_URL aponta para `/railway`, não `/postgres`)
- **Deploy** via `railway up --detach` (manual, Railway não está conectado ao GitHub para auto-deploy)
